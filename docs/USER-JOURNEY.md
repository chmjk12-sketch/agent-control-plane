# Agent 全生命周期用户旅程 V2

> 日期: 2026-06-15
> 版本: v2.0 (融入 5 项改进)

---

## 一、5 项核心改进

| # | 改进 | 说明 |
|---|------|------|
| 1 | **模板自我迭代** | 模板仓库根据使用情况自动优化，经用户确认后更新 |
| 2 | **UI 仿照行业顶尖** | 参考 Dify/Coze/n8n 的 UI 设计，从用户需求出发 |
| 3 | **准备阶段全自动** | 已跑通的部分全自动完成，不让用户参与 |
| 4 | **本地验收后上线** | 代码上线前必须在本地通过完整验收 |
| 5 | **网址生成需确认** | Agent 的访问地址需要用户确认后生效 |

## 二、TRAE 如何知道 Agent 规范

### 2.1 规范传递机制

```
┌─────────────────────────────────────────────────────────────────┐
│                    Agent 模板仓库 (GitHub)                         │
│                                                                 │
│  agent-template/                                                │
│  ├── .trae/                                                      │
│  │   └── rules.md          ← TRAE 自动读取此文件               │
│  ├── agent.yaml              ← Agent 元数据                      │
│  ├── Dockerfile              ← 标准化构建                        │
│  ├── docker-compose.yml      ← 本地开发                          │
│  ├── .github/workflows/                                              │
│  │   └── deploy.yml          ← CI/CD（复用控制平面）              │
│  ├── src/                                                         │
│  │   ├── main.py             ← FastAPI 入口模板                   │
│  │   ├── api/v1/             ← API 路由模板                      │
│  │   ├── models/             ← Pydantic 模型模板                  │
│  │   └── config.py           ← 配置模板                          │
│  ├── frontend/               ← 可选前端模板                       │
│  └── tests/                  ← Mock 测试模板                      │
│                                                                 │
│  ★ 模板自我迭代机制:                                             │
│  - 控制平面收集所有 Agent 的构建/部署/运行数据                     │
│  - 每月分析: 哪些步骤经常失败、哪些配置经常需要手动修改              │
│  - 自动生成模板改进建议 (如: "90% 的 Agent 都需要 /health 端点")    │
│  - 经用户确认后更新模板仓库                                       │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 模板自我迭代流程

```
控制平面收集数据
  │
  ├─→ CI/CD 失败率统计 (哪些步骤经常失败)
  ├─→ 手动修复频率 (哪些配置经常需要人工干预)
  ├─→ Agent 运行时错误 (哪些依赖/版本经常出问题)
  └─→ 用户反馈 (哪些模板不够好用)
        │
        ▼
  自动生成改进建议
  │  例: "近 30 天 5 个 Agent 在 Prisma 迁移步骤失败，
  │       建议在模板中增加迁移前检查脚本"
  │
  ▼
  控制平面 UI 展示建议
  │  "发现 3 个模板改进建议，是否查看？"
  │
  ▼
  用户确认 (或拒绝)
  │
  ├─→ 确认: 自动更新模板仓库 + 通知所有 Agent 项目
  └─→ 拒绝: 记录原因，下次不再提示
```

### 2.3 三种创建方式

| 方式 | 操作 | 适用场景 |
|------|------|----------|
| **方式 A：控制平面一键创建** | UI 填表 → 自动 fork 模板 → 生成 repo → 自动部署 | 批量生产 |
| **方式 B：模板仓库 Clone** | `git clone agent-template my-agent` → TRAE 打开 → 开发 | 新建 Agent |
| **方式 C：存量项目接入** | 存量项目 → 添加 `agent.yaml` + `.trae/rules.md` → 推送 | 已有项目 |

## 四、用户全生命周期旅程（改进版）

### 阶段 0：全自动准备（用户无需参与）

```
系统自动完成 (一次性，已在 V2.0 中跑通)
  │
  ├─→ ✅ GitHub 组织已创建 (chmjk67-sketch)
  ├─→ ✅ ACR 镜像仓库已配置
  ├─→ ✅ ECS 服务器已部署 (39.105.86.184)
  ├─→ ✅ PostgreSQL 已运行
  ├─→ ✅ Caddy 反向代理已配置
  ├─→ ✅ GitHub Actions CI/CD 已跑通
  ├─→ ✅ 控制平面已上线
  │
  └─→ ★ 新增 Agent 时自动完成:
        ├─→ Fork 模板仓库
        ├─→ 创建 GitHub repo
        ├─→ 配置 CI/CD (复用控制平面 workflow)
        ├─→ 注册到控制平面
        └─→ 生成 Caddy 路由
```

**用户零操作**。所有准备工作已跑通，新 Agent 创建时全自动完成。

### 阶段 1：创建 Agent（控制平面 UI）

```
用户 (开发者)
  │
  ├─→ 打开控制平面 → 点击 "+ 新建 Agent"
  │
  ├─→ 填写向导 (4 步):
  │     ├─→ Step 1: 基本信息 (名称、描述、模板类型)
  │     ├─→ Step 2: AI 配置 (模型、温度、系统提示)
  │     ├─→ Step 3: 访问配置 (★ 用户确认网址)
  │     └─→ Step 4: 确认创建
  │
  └─→ 系统自动完成:
        ├─→ Fork 模板 → 创建 repo → 配置 CI/CD
        ├─→ 返回: repo URL + 本地启动命令
        └─→ 提示: "在 TRAE 中打开项目开始开发"
```

### 阶段 2：开发 Agent（TRAE + 本地验收）

```
用户 (开发者) 在 TRAE 中打开项目
  │
  ├─→ TRAE 自动读取 .trae/rules.md
  │
  ├─→ 开发业务逻辑
  │     ├─→ TRAE 按规范生成代码
  │     ├─→ 自动实现 /health、/metrics
  │     └─→ 自动生成 Mock 测试
  │
  ├─→ ★ 本地验收 (上线前必须通过)
  │     ├─→ docker-compose up (本地启动)
  │     ├─→ pytest tests/ (单元测试，100% 通过)
  │     ├─→ curl http://localhost:8000/health (健康检查)
  │     ├─→ curl http://localhost:8000/api/v1/analyze (业务测试)
  │     └─→ 检查控制平面集成:
  │           ├─→ 验证 /metrics 指标输出
  │           └─→ 验证 MCP 工具暴露 (如启用)
  │
  ├─→ ★ 验收通过 → 推送到 GitHub
  │     └─→ git push origin main
  │
  └─→ ★ 验收未通过 → 修复后重新验收
        └─→ TRAE 辅助定位问题
```

### 阶段 3：自动部署（CI/CD + 网址确认）

```
GitHub 收到 push
  │
  ├─→ GitHub Actions 自动触发
  │     ├─→ Step 1: pytest tests/ (Mock 测试)
  │     ├─→ Step 2: Docker Build
  │     ├─→ Step 3: Push to ACR
  │     └─→ Step 4: SSH 部署到 ECS
  │           ├─→ 拉取镜像
  │           ├─→ 启动容器 (blue 环境)
  │           ├─→ 健康检查
  │           └─→ 通知控制平面
  │
  └─→ 控制平面收到部署通知
        ├─→ 解析 agent.yaml
        ├─→ 创建 Agent 记录
        ├─→ ★ 生成访问地址 → 推送给用户确认
        │     └─→ 用户确认后:
        │           ├─→ 生成 Caddy 路由片段
        │           ├─→ caddy reload (秒级生效)
        │           └─→ 开始健康检查轮询
        └─→ 飞书/钉钉通知: "Agent xxx 已部署成功"
```

### 阶段 4：日常使用

```
用户 (终端用户)
  │
  ├─→ 控制平面 UI
  │     ├─→ Agent 列表 (仿 Dify 应用列表)
  │     ├─→ 点击 Agent → iframe 嵌入前端 (仿 Dify 运行界面)
  │     └─→ 实时对话/调试 (仿 Coze 对话界面)
  │
  ├─→ API 网关调用
  │     ├─→ POST /api/proxy/{slug}/analyze
  │     └─→ 自动: 鉴权 → 限流 → 代理 → 记录 Token → 计费
  │
  └─→ 多 Agent 协同
        ├─→ 工作流编排 (仿 n8n 拖拽)
        └─→ 动态协商 (Agent 间消息总线)
```

### 阶段 5：运维监控

```
用户 (运维/管理员)
  │
  ├─→ 监控大盘 (仿 Grafana)
  │     ├─→ Agent 状态卡片 (在线/离线/降级)
  │     ├─→ 请求量、Token、成本趋势图
  │     └─→ 实时日志流 (仿 Vercel)
  │
  ├─→ 成本管理
  │     ├─→ 月度预算设置
  │     ├─→ 超支自动拦截
  │     └─→ 成本报表
  │
  ├─→ 版本管理 (仿 Vercel)
  │     ├─→ 版本历史列表
  │     ├─→ 灰度发布 (流量权重滑块)
  │     └─→ 一键回滚
  │
  └─→ 告警中心
        ├─→ 健康告警 (Agent 宕机)
        ├─→ 成本告警 (预算超限)
        └─→ 部署告警 (CI/CD 失败)
```

### 阶段 6：迭代更新

```
用户 (开发者)
  │
  ├─→ TRAE 修改代码
  ├─→ 本地验收通过
  ├─→ 推送到 GitHub
  └─→ 自动: CI/CD → 蓝绿部署 → 零停机更新
        └─→ 飞书通知: "Agent xxx 已更新到 v1.2"
```

## 六、用户交互点清单

| 阶段 | 用户操作 | 自动完成 |
|------|----------|----------|
| 0. 准备 | **无** | GitHub 组织、ACR、ECS、CI/CD 全自动 |
| 1. 创建 | 填写名称/描述、选模板、**确认网址** | Fork、创建 repo、配置 CI/CD |
| 2. 开发 | 在 TRAE 中描述需求、**本地验收** | 代码生成、Mock 测试 |
| 3. 部署 | **确认访问地址生效** | CI/CD、Docker Build、蓝绿部署 |
| 4. 使用 | 通过 UI/API 使用 Agent | 鉴权、计费、监控 |
| 5. 运维 | 查看大盘、处理告警 | 健康检查、自动告警 |
| 6. 迭代 | TRAE 修改、**本地验收** | 自动部署、零停机更新 |

**用户只需在 3 个环节确认：创建时确认网址、开发后本地验收、部署后确认生效。**
